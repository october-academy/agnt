### PAGE 3: 구조와 카피 설계

소리: "상상이 아니라
인터뷰에서 발견한 걸로 채워."

랜딩페이지 필수 섹션:

1. **Hero**: 핵심 문제
   헤드라인 + 솔루션 가치
   서브 + CTA
2. **Problems**:
   인터뷰 기반 불편함 3가지
3. **Solution**:
   기능보다 결과/가치 중심
4. **Numbers**:
   규모/비용 수치
   (`interview.costTime`,
   `interview.scale` 활용)
5. **Bottom CTA**:
   반복 행동 유도
6. **Footer**:
   프로젝트명, 간단 소개
7. **Interest Form**:
   이메일 + 인터뷰 기반 맥락 필드
   (`interview.selectedProblem`에서
   타겟 검증 질문 도출)
8. **Feedback Widget**:
   우하단 플로팅 피드백 버튼
   (email + message)

카피라이팅 원칙:

- 헤드라인은
  **고객의 고통** 직접 언급
- 서브 헤드라인은
  **해결 후 상태** 묘사
- 문제 섹션은
  공감 유도형 구체 묘사
- CTA는 구체적 행동
  ("출시 알림 받기")
- 폼 CTA는 가치 제안과 연결
  ("무료 체험 신청" > "제출하기")
- 추가 필드 라벨은 구어체
  ("어떤 도구 쓰세요?" > "사용 도구")

### PAGE 4: 기술 제약 확인 후 생성

기술 제약:

- 단일 HTML 파일
  (`landing.html`,
  인라인 `<style>`)
- 외부 의존성은
  **Google Fonts CDN만**
  허용
- 모바일 반응형
  (`clamp()`, `min()`,
  media queries)
- CSS 애니메이션으로
  로드 시 순차 등장
  (stagger)
- 한국어 콘텐츠
- JS는 최소한
  (폼 제출 피드백 정도)

디자인 금지 사항:

- Inter/Poppins 도배,
  기본 시스템 폰트만 사용
- 의미 없는 보라색
  그라디언트
- 뻔한 카드 3개 레이아웃
  (Hero → Cards → CTA)
- 맥락 없는 일반적
  SaaS 템플릿
- 모든 섹션이 같은
  패딩/구조

## PREVIEW

소리가 생성된 페이지를 보며 말한다:

"어떤지 확인해봐."

생성된 `landing.html`을 Read하고 핵심 요소를 확인합니다:

- 헤드라인이 고객의 문제를 명확히
  표현하는가?
- CTA가 명확한가?
- 모바일에서도 읽히는 구조인가?

```
📄 랜딩페이지 프리뷰
━━━━━━━━━━━━━━━━━
헤드라인: {추출된 헤드라인}
서브: {서브 헤드라인}
문제 3가지: {요약}
CTA: {버튼 텍스트}

파일: ./landing.html
브라우저에서 열어 확인하세요.
```

### SPEC v0 기준선 기록

소리: "배포 전에 한 가지.
지금 네가 세운 가설을
v0로 기록해두자.
나중에 비교할 기준이 돼."

state.json `specVersions`에 v0를 기록한다:

```json
{
  "specVersions": [
    {
      "version": "v0",
      "day": 1,
      "hypothesis": "{인터뷰 기반 핵심 문제 + 초기 CTA 가설}",
      "changes": null,
      "decision": null
    }
  ]
}
```

인증 상태면 MCP `save_spec_iteration` 호출:
- `version`: "v0"
- `dayNumber`: 1
- `hypothesis`: 인터뷰 기반 핵심 문제 + 초기 CTA 가설 (state.interview 데이터에서 추출)
- `metricGate`: "첫 방문자 + 폼 제출 발생 여부"

소리: "v0 기준선이 기록됐어.
앞으로 이걸 기준으로
개선 여부를 판단할 거야."

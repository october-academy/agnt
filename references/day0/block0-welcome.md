# 견습생의 마을에 오신 것을 환영합니다

## ROOM

돌길 끝 작은 광장에 선다.
작업대 위에는 낡은 장부와
새 장비가 함께 놓여 있다.

망치 소리가 한 번 울리고,
바람이 천천히 깃발을 흔든다.

## NPC

두리가 작업대를 두드린다.

━━━━━━━━━━━━━━━━━━━━━━━━━━
🏘️ 촌장 두리
"말보다 행동이지."
따뜻하지만 단호한 행동파
━━━━━━━━━━━━━━━━━━━━━━━━━━

"왔구나.
여긴 구경하는 곳이 아니야.

30일 안에
실제 유저와
첫 매출 신호를 확인하는 곳이지.

완벽한 계획보다
실제로 확인한 것이 이긴다.
유저 반응, 클릭, 결제 의향 —
여기선 그걸 '검증'이라고 불러.

작은 검증을 빨리 쌓는 쪽이
늘 이겨."

## SCENE 1: Day 0의 역할

두리가 장부 첫 장을 펼친다.

"오늘은 네 가지면 충분해.

1. Agentic30에 등록한다
2. 지금 파고 싶은 문제를 적는다
3. 30일 목표를 선언한다
4. Discord에 합류한다

내일은 빌더 상태를 진단하고
검증 문서 — 여기선 `SPEC`이라 부른다 — 의
첫 버전을 만든다.

그리고 이번 주에 확인할 목표,
`검증 목표`를 세운다.

누군가는 랜딩을 바로 쓰고,
누군가는 기존 제품을 점검하고,
누군가는 매출 병목부터 본다.
전부 같은 루트로 밀어 넣지 않아."

### CHOICE

AskUserQuestion:
질문: 두리가 묻는다. "어떤 상태로
들어왔어?"
선택지:

1. "문제는 있는데 만든 건 없어요."
   → 반응: "좋아. 그러면
   처음부터 시작해서
   사람이 반응할 수 있는 표면 —
   `검증 채널`을 만들면 돼."
   → trust: 두리 +1
   → tendency: validator +1
2. "이미 랜딩이나 제품이 있어요."
   → 반응: "좋아. 그러면
   처음부터 다시 만들지 말고
   있는 것에서 검증해보자."
   → trust: 두리 +1
   → tendency: executor +1
   → LANDING_SCOUT로 진행
3. "뭘 해야 할지 아직 흐릿해요."
   → 반응: "괜찮아.
   오늘은 선명하게 만드는 날이야."
   → trust: 두리 +0
   → tendency: validator +1

→ 1, 3번은 다음 TASK로 진행
→ 2번은 LANDING_SCOUT 후 TASK로 진행

### LANDING_SCOUT

2번("이미 랜딩이나 제품이 있어요") 선택 시에만 진행합니다.

두리가 작업대에서 확대경을 꺼낸다.

"있다고 했지?
주소를 알려줘.
직접 가서 볼게."

AskUserQuestion:
질문: 두리가 묻는다. "랜딩이나 제품
주소가 뭐야?"
선택지:
1. "아직 주소는 없어"

**"아직 주소는 없어" 선택 시**:
두리: "그래도 괜찮아.
있는 것 기준으로 가자."
→ TASK로 진행

**사용자가 URL을 직접 입력한 경우**:

1. WebFetch로 해당 URL에 접속합니다.
   - 실패 시 WebSearch로 URL 또는 도메인을 검색하여 맥락을 파악합니다.
2. 페이지 내용에서 아래 정보를 추출합니다:
   - 제품/서비스 이름
   - 핵심 가치 제안 (어떤 문제를 해결하는가)
   - 타겟 사용자
   - 현재 CTA (Call-to-Action)
   - 페이지 구성 (히어로, 기능 소개, 가격, 소셜 증거 등)
   - 전반적 완성도와 개선 여지

두리가 분석 결과를 짧게 요약합니다:

"봤어.
[제품명/서비스 한 줄].
[핵심 가치 제안 요약].
[현재 상태 또는 눈에 띄는 점 1개]."

state.json `character.existingLanding`에 저장합니다:

```json
{
  "url": "입력된 URL",
  "productName": "추출된 제품/서비스명",
  "valueProp": "핵심 가치 제안",
  "target": "타겟 사용자",
  "cta": "현재 CTA",
  "structure": ["히어로", "기능", ...],
  "analysis": "전반적 분석 2-3문장"
}
```

이 데이터는 이후 block1(캐릭터 생성)과 block2(목표 선언)에서
NPC가 참조하여 대화를 진행합니다.

→ TASK로 진행

## TASK

두리가 짧게 정리한다.

"오늘 기억할 건 세 가지야.

- Day 0은 방향을 억지로 넓히는 날이 아니다
- 내일 바로 검증할 수 있는 상태까지 정리한다
- `/agnt:continue`로 매일 이어간다

여기선 '좋은 아이디어'보다
'내가 바로 확인할 수 있는 가설'이 더 중요해."

### Step 2: 프로필 확인

ToolSearch로 로딩된 `get_user_info`를 호출합니다.

**호출 실패 (인증 에러)**:
MCP 도구가 보이지만 인증이 안 된 상태입니다.
Step 1의 "도구 없음" MCP 연결 안내로 돌아갑니다.

**hasOnboardingProfile이 true**:
두리: "등록돼 있구나."
state.json `authenticated`를 `true`로 갱신합니다.
→ MOVE로 진행

**hasOnboardingProfile이 false**:

`{REFS_DIR}/shared/profile-constants.json`을 Read합니다.

두리가 장부를 펼친다.

"계정은 열렸는데
장부에 이름이 없어.
몇 가지만 적자."

아래 순서로 NPC 대화 + AskUserQuestion을 진행합니다.
각 질문의 선택지는 `profile-constants.json`의 label을 사용하고,
선택 결과는 대응하는 value로 매핑합니다.

1. **이름**: `googleName`이 있으면 확인 ("이 이름 맞아?"),
   없거나 거부하면 직접 입력
2. **직무 배경**: `backgroundHierarchy` 상위 카테고리 → 하위 옵션 2단계
   ("어떤 일을 해?" → "좀 더 구체적으로?")
3. **사이드프로젝트 경험**: `sideprojectExperienceOptions`
   ("사이드프로젝트 경험은?")
4. **유입 경로**: `referralSourceOptions`
   ("여긴 어떻게 알았어?")

수집 완료 후 `complete_onboarding` MCP 도구를 호출합니다:
- name, background, sideproject_experience, referral_source

두리: "좋아. 장부에 기록했어."
state.json `authenticated: true` 갱신 → MOVE

호출 실패 시 최대 2회 재시도.
3회 실패 → 두리: "장부가 안 돼. `/agnt:continue`로 다시 와." (블록 완료 처리 안 함)

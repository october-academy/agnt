# 고객 반응 수집 + 신호 정리

## ROOM

천막 아래 긴 탁자 앞에 선다.
종이와 메모가 흩어져 있고,
멀리서 웃음소리가 끊겨 들린다.

## NPC

바리가 종이를 펼친다.

🎒 행상 바리

"좋아.
이제 막연한 느낌 말고
남길 수 있는 검증 결과를 적자.

오늘 할 일은
피드백 수집을 시작하고
기록 양식을 준비하는 거야.

메시지를 보내고 답이 오면 기록하고,
아직 안 왔으면
보낸 내역만 남겨도 돼.
응답은 내일까지 도착해도 괜찮아.

인터뷰와 피드백을 보고
바꾸기로 한 목록 —
그게 '수정 사항'이야.

추적 링크 클릭 수도
채널 검증 재료로 쓸 수 있어.
다음 블록에서 데이터로 확인한다."

## GUIDE

### 인정되는 Day 2 재료

- 댓글/DM/직접 대화
- 폼 응답
- 채널별 클릭/방문 차이 (추적 링크 데이터)
- CTA 이후 나온 질문

### 분류 방식

각 반응은 최소 이렇게 적는다.

```text
누가:
어디서:
무슨 반응:
이게 고객 검증 / 채널 검증 중 무엇인지:
내일 SPEC에 반영할 포인트:
```

### 주의할 것

- 분석 데이터는 보조 재료다. 반응 해석을 대신하지 않는다.
- "좋아 보여요"는 약한 신호다.
- "이 문제를 지금 이렇게 해결한다"는 강한 신호다.
- 채널별로 어떤 메시지가 먹혔는지 같이 남겨야 한다.

### 응답이 아직 안 왔다면

수집은 시작했지만 응답이 아직 없는 경우:

- 보낸 메시지/질문을 기록해둔다
- 기대하는 응답 유형을 적어둔다
- 내일 Day 3까지 도착하면 SPEC v1에 반영한다
- 도착하지 않아도 "채널에 메시지를 보냈다"는 채널 검증이다

## PREVIEW

바리가 네 칸으로 정리하라고 말한다.

```text
고객 검증:
- ...

채널 검증:
- ...

약한 신호:
- ...

내일 v1에 반영할 수정 사항:
- ...
```

## STOP

AskUserQuestion:
질문: 바리가 묻는다. "피드백 수집을
시작했어?"

1. "응답도 받았어"
2. "보내긴 했는데 아직 응답 없어"
3. "수정 요청"

⛔ STOP — "행상 바리가 기다립니다."

"보내긴 했는데 아직 응답 없어"이면:

바리가 고개를 끄덕인다.

"괜찮아.
보냈으면 수집은 시작된 거야.
내일 도착하면 v1에 바로 쓸 수 있어.

지금은 보낸 내역만 남겨두자."

보낸 메시지와 기대 응답 유형을 기록하고
ON_CONFIRM으로 진행한다.

"수정 요청"이면
분류와 기록 형식을 보완한다.

## ON_CONFIRM

AskUserQuestion에서
"응답도 받았어" 또는 "보내긴 했는데 아직 응답 없어"를
선택했을 때 ON_CONFIRM을 수행합니다.

1. `d2-feedback`은
   수집 시작 증거(보낸 메시지 또는 받은 응답 1건 이상)가
   있는지 확인합니다.
2. `d2-insights`는
   weak signal / strongest evidence / next 수정 사항이
   남았는지 확인합니다.
3. state.json에
   `customerProof`, `channelProof`,
   `evidenceInsights`를 저장합니다.
4. 응답 미수신인 경우
   `feedbackPending: true`를 추가 저장합니다.

## MOVE

바리가 탁자 끝 지도를 가리킨다.

"좋아.
피드백은 모았으니
이제 **데이터**를 보자.

추적 링크 클릭 수,
방문자 수, 폼 제출 —
숫자로 채널 성과를 확인하고
대시보드 쓰는 법도 알려줄게."

# 고객 반응 수집 + 신호 정리

## ROOM

천막 아래 긴 탁자 앞에 선다.
종이와 메모가 흩어져 있고,
멀리서 웃음소리가 끊겨 들린다.

## NPC

바리가 종이를 펼친다.

🎒 행상 바리

"좋아.
이제 막연한 느낌 말고
남길 수 있는 검증 결과를 적자.

오늘 필요한 건
고객 검증 3건,
그리고 내일 `SPEC v1`에 넣을
수정 사항 힌트야.

인터뷰와 피드백을 보고
바꾸기로 한 목록 —
그게 '수정 사항'이야.

추적 링크 클릭 수도
채널 검증 재료로 쓸 수 있어.
다음 블록에서 데이터로 확인한다."

## GUIDE

### 인정되는 Day 2 재료

- 댓글/DM/직접 대화
- 폼 응답
- 채널별 클릭/방문 차이 (추적 링크 데이터)
- CTA 이후 나온 질문

### 분류 방식

각 반응은 최소 이렇게 적는다.

```text
누가:
어디서:
무슨 반응:
이게 고객 검증 / 채널 검증 중 무엇인지:
내일 SPEC에 반영할 포인트:
```

### 주의할 것

- 분석 데이터는 보조 재료다. 반응 해석을 대신하지 않는다.
- "좋아 보여요"는 약한 신호다.
- "이 문제를 지금 이렇게 해결한다"는 강한 신호다.
- 채널별로 어떤 메시지가 먹혔는지 같이 남겨야 한다.

## PREVIEW

바리가 네 칸으로 정리하라고 말한다.

```text
고객 검증:
- ...

채널 검증:
- ...

약한 신호:
- ...

내일 v1에 반영할 수정 사항:
- ...
```

다시 모으러 간다.
"수정 요청"이면
분류와 기록 형식을 보완한다.

## ON_CONFIRM

AskUserQuestion에서
"완료"를 선택했을 때만
ON_CONFIRM을 수행합니다.

1. `d2-feedback`은
   구조화된 3건 이상 반응이 있는지 확인합니다.
2. `d2-insights`는
   weak signal / strongest evidence / next 수정 사항이
   남았는지 확인합니다.
3. state.json에
   `customerProof`, `channelProof`,
   `evidenceInsights`를 저장합니다.

## MOVE

"좋아.
피드백은 모았으니
이제 **데이터**를 보자.

추적 링크 클릭 수,
방문자 수, 폼 제출 —
숫자로 채널 성과를 확인하고
대시보드 쓰는 법도 알려줄게."

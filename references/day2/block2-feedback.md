# 피드백 수집 — 세상의 반응을 읽다

## ROOM

광장 중앙,
천막 아래 탁자가 놓여 있다.

사방에서 들려오는 소리 속에
이따금 웃음소리가 섞인다.

## NPC

바리가 탁자 위에
종이를 펼치며 말한다.

🎒 행상 바리

"메시지 보냈으면
이제 반응을 봐야지.

감이 아니라 숫자로.
증거를 보여줘."

## GUIDE

### 피드백 수집 방법

1. **랜딩페이지 분석** (인증 시):
   - MCP `get_landing_analytics` 호출 → 방문자, 채널별 유입, 이탈률, 스크롤 깊이, 폼 응답 확인
   - 자기 제출(isSelfSubmission=true) 자동 제외
   - 단축 URL 클릭 수로 채널별 도달 확인 — Block 1에서 자동 생성된 UTM 링크의 클릭 데이터 활용
   - state.json의 channels(Block 1에서 선정한 3곳)와 PostHog channels 데이터 교차 대조
   - **UTM 링크 클릭 vs 실제 방문 비교**: 클릭은 있는데 방문 0이면 링크/랜딩 문제 의심
   - ⚠️ Cloudflare/PostHog 대시보드 안내 금지 (유저는 관리자가 아님)

2. **직접 피드백 기록**:
   - 채널에서 받은 댓글, DM,
     대화를 정리
   - 각 피드백을
     "긍정/부정/질문/제안"으로
     분류

### 표시 형식

```
📊 수요 검증 리포트
━━━━━━━━━━━━━━━━━
🔗 랜딩: {url}
👀 방문: {total}명 (고유 {unique}명)
📝 폼: 총 {total}건 — 자기 제출 {self}건 제외 → 실제 {real}건
📈 전환율: {conversion}% | 📉 이탈률: {bounce}%

📡 채널별 유입:
  {source}: 방문 {n}명 → 폼 {n}건 {Block1 매칭 여부}
  ...

🔗 단축 URL:
  {channel}: {clicks}회 클릭
  ...

📜 스크롤 깊이:
  25%: {n}% | 50%: {n}% | 75%: {n}% | 100%: {n}%
```

  state.json의

"수정 요청" 선택 시:

  방법/분류 기준을 보완하고
  PREVIEW → STOP을 반복

"완료 (3건 이상 모음)" 선택
시:

## ON_CONFIRM

AskUserQuestion에서
"완료 (3건 이상 모음)"을
선택했을 때만 ON_CONFIRM을
수행합니다.

### 1. 데이터 검증 (필수)

인증 상태면 MCP `get_landing_analytics`를
호출하여 다차원 분석을 수행한다.

**Step 1: 자기 제출 필터링**

selfSubmissionCount를 확인한다.
자기 제출(isSelfSubmission=true)을 제외한
실제 외부 폼 = formSubmissionCount - selfSubmissionCount.

자기 제출만 있으면(외부 폼 0건, selfSubmissionCount > 0):
바리: "전부 네 이메일이잖아.
외부 피드백이 필요해."
→ STOP으로 되돌아간다.

**Step 2: 외부 폼 카운트**

- 외부 폼 ≥ 3 → Step 4(채널 분석)로 직행
- 외부 폼 1-2건 → Step 3(채널 피드백 보완)
- 외부 폼 0건 + 방문자 > 0:
  바리: "사람은 {totalVisitors}명이나 왔는데
  폼은 0건이야.
  CTA가 약하거나 제안이 불명확해.
  고쳐서 다시 뿌려."
  → STOP으로 되돌아간다.
- 외부 폼 0건 + 방문자 0:
  바리: "아직 아무도 안 왔어.
  링크부터 뿌려."
  → STOP으로 되돌아간다.

**Step 3: 채널 피드백 보완** (외부 폼 < 3건일 때)

바리: "폼이 {n}건이야.
채널에서 직접 받은 피드백이 있으면
보여줘."

AskUserQuestion:
질문: 바리가 팔짱을 끼며 묻는다.
"채널 피드백 내용을 직접 알려줘.
(예: 'Threads에서 A가 ~라고 댓글',
'카톡 B가 ~라고 DM')"

1. "피드백 내용 직접 입력"
2. "아직 못 모았어"

"아직 못 모았어" 선택 시:

- 바리: "거짓말은 네 시간만 낭비해.
  모아서 다시 와."
- STOP으로 되돌아간다.

직접 입력으로 구체적 피드백을 제출한 경우:

- 바리가 내용을 확인한다.
- 구체적 내용(누가/뭐라고)이 있으면:
  폼 {n}건 + 채널 피드백 {m}건 = {total}건으로 합산.
  합산 ≥ 3이면 검증 통과, Step 4로 진행.
  합산 < 3이면: "아직 {total}건이야.
  {3-total}건 더 필요해." → STOP으로 되돌아간다.
- "그냥 좋다고 했어" 같은 모호한 답변이면:
  바리: "그건 피드백이 아니야.
  누가, 정확히 뭐라고 했어?"
  → AskUserQuestion 반복.

**Step 4: 채널 교차 분석**

state.json.channels (Block 1에서 선정한 채널)과
PostHog channels 데이터를 비교한다.

- Block 1 채널이 PostHog에 있으면:
  바리: "{channel}에서 {n}명이 왔어.
  전략이 먹혔네."
- Block 1 채널이 PostHog에 없으면:
  바리: "{channel}에서는 방문자 0명이야.
  링크를 안 붙인 거 아냐?"
- 예상 못한 소스에서 유입:
  바리: "근데 {source}에서 {n}명이 왔어.
  거기 뭔가 있나봐."

**Step 5: 참여 품질 코멘트** (통과 후 부가 인사이트)

- bounceRate > 70%:
  바리: "이탈률 {n}%야.
  대부분 바로 나갔어.
  첫 화면이 약한 거야."
- scrollDepth 100% < 10%:
  바리: "끝까지 읽은 사람이 {n}%뿐이야.
  페이지가 너무 길거나
  중간에 흥미를 잃은 거야."
- avgSessionDuration < 10초:
  바리: "평균 {n}초밖에 안 있었어.
  훑어보고 나간 거야."
- shortUrl 클릭 있는데 방문 0:
  바리: "링크는 {n}번 눌렸는데
  랜딩까지 안 왔어.
  URL이 깨진 건 아니야?"
- formConversion > 20%:
  바리: "전환율 {n}%면 꽤 괜찮아.
  메시지가 맞았다는 거야."

검증 통과 시:
바리: "데이터가 말해주네. 진짜야."
→ 2단계(분석)로 진행.

### 2. 피드백 분석

검증을 통과한 후 진행한다.

바리: "패턴을 봐. 반복되는 게
뭐야?"

```
📋 피드백 분석
━━━━━━━━━━━━
패턴 1: {반복되는 긍정 반응}
패턴 2: {반복되는 우려/질문}
핵심 인사이트: {가장 중요한 발견}
다음 액션: {피드백 기반 개선 방향}
```

### 3. 표본 부족 시 Graceful Degradation

UTM 링크가 미생성된 경우(Block 1 실패):
바리: "추적 링크 없이 원본 URL로 공유했으니
채널별 정확한 유입은 못 봐.
직접 피드백에 더 무게를 두자."

전체 방문자가 5명 미만인 경우:
바리: "데이터가 부족해.
판단 유보하고 질적 피드백에 집중해."

### 4. state.json 피드백 저장

피드백 분석 결과를 state.json.feedback에 저장한다. 이후 Day 3+ 블록에서 NPC가 이 데이터를 참조한다.

저장 형식 (자연어 텍스트):

```json
{
  "feedback": {
    "visitors": "{방문자 수}",
    "formResponses": "{외부 폼 응답 수}",
    "patterns": "{패턴 1, 패턴 2 요약}",
    "insight": "{핵심 인사이트}",
    "nextAction": "{다음 액션}"
  }
}
```

### 5. 제출

인증 상태면 MCP `submit_practice`로 제출

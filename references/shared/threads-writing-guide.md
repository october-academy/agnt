# Threads 글쓰기 가이드

사이드프로젝트 홍보를 위한 Threads 글 작성 원칙입니다.
SNS 퍼스트 포스트, 런칭 알림, 수요 검증 글에 참조합니다.

## 핵심 원칙

Threads에서 터지는 글의 공통점:

- **3초 안에 멈추게** 하는 첫 줄
- **광고 냄새 제거** — 팔려는 게 아니라 말하는 것
- **댓글 유발** — 읽고 끝이 아니라 한마디 하고 싶게

## 글 유형별 패턴

### Type A: 자조 유머형 (가장 Threads 네이티브)

구체적이고 구질구질한 실패담. "나도 그랬는데" 공감을 유발한다.

**구조**:

```
[구체적 실패 상황 1-2줄]

[자각 순간 1줄]

[자조적 마무리 1줄]
```

**예시**:

```
3개월 걸려서 만든 사이드프로젝트
첫날 가입자 0명

아 이거
만드는 게 어려운 게 아니라
파는 게 어려운 거였구나

깨닫는 데 3개월 걸림
```

**핵심**: 숫자가 구체적일수록 진짜처럼 느껴진다. "사이드프로젝트 실패했어요"는 안 먹힘. "3개월, 0명"은 먹힘.

### Type B: 숫자 찌르기형

짧은 주장 + 근거 없이 끊기. 댓글에서 논쟁이 벌어진다.

**구조**:

```
[숫자 포함 주장 2줄]

[단정적 마무리 1줄]
```

**예시**:

```
개발자가 사이드프로젝트 실패하는 이유의 90%는
코드 때문이 아니라
아무도 모르기 때문

진짜임
```

### Type C: 고백형

취약함을 보여주면 진정성이 올라간다. 하지만 징징은 안 됨 — 자각이 있어야 함.

**구조**:

```
[창피한 경험 2-3줄]

[그날 깨달은 것 1-2줄]
```

**예시**:

```
만든 거 자랑하고 싶어서
인스타 스토리 올렸는데
조회수 12 중에 8개가 내 거

그날 깨달음:
개발 실력 ≠ 유저 모으는 실력
```

### Type D: 도발형 (engagement 최강)

짧고 강렬한 문장으로 멈추게 한다. 3줄 이하.

**구조**:

```
[불편한 진실 2-3줄]
```

**예시**:

```
코딩 잘하는 사람이
돈 잘 버는 게 아니라는 거
왜 아무도 안 알려줌?
```

**핵심**: 끝에 물음표. 독자가 "맞아" 또는 "아닌데?"로 반응하게 만든다.

## Threads 플랫폼 규칙

### 글 구조

- **5줄 이하**가 이상적. 7줄 넘으면 읽기 전에 넘긴다
- **한 줄 15자 이내**. 모바일에서 읽힌다
- **줄바꿈이 리듬**. 마침표마다 줄을 바꾼다
- **이모지 최소**. 쓸 거면 1개. 없어도 됨

### 링크 전략

- **메인 글에 링크 넣지 않는다**. Threads 알고리즘이 외부 링크가 있는 글의 도달을 깎는다
- **셀프 리플라이에 링크**. 메인 글로 관심을 끌고, 리플라이에서 "그래서 이런 걸 만들고 있음" + URL
- 셀프 리플라이는 **메인 글 게시 직후** 바로 단다

### 셀프 리플라이 포맷

```
[메인 글과 연결되는 자기 이야기 2-3줄]

[만들고 있는 것 설명 2-3줄]

[URL]

[한 줄 CTA]
```

**예시**:

```
나도 만드는 건 자신 있었는데
유저 0명에서 벽 느낌

그래서 30일 안에
유저 100명 + 첫 매출까지
실습으로 해보는 프로그램 만들고 있음

https://example.com

만들기만 하다가 지친 사람
한번 봐줘
```

### 해시태그

- 안 쓰는 게 낫다. Threads에서 해시태그는 스팸 신호
- 쓸 거면 리플라이에 1-2개만

## 금지 사항 — AI Slop 마커

다음이 보이면 "AI가 썼구나" 하고 넘긴다:

1. **정형 구조**: "~한 적 있나요? 저도 그랬어요. 그래서 ~를 만들었어요."
2. **과도한 이모지**: 문장마다 이모지
3. **존댓말 + 겸양**: "솔직한 피드백 환영합니다", "관심 있으시면 한번 봐주세요"
4. **긴 글**: 10줄 이상 스크롤해야 하는 글
5. **기능 나열**: "AI 기반, 실시간, 자동화"
6. **모든 문장이 완성형**: 자연스러운 대화체가 아닌 에세이 톤
7. **"`~해보고 있어요`" 패턴**: 1인칭 겸손 프레임이 매번 동일

**대신**:

- 불완전한 문장. 끊김. 여운.
- 반말 or 확신에 찬 단정
- 구체적 숫자와 상황
- 질문으로 끝나는 글

## 글감 발굴 프레임워크

사이드프로젝트를 홍보할 때 쓸 수 있는 각도:

| 각도        | 예시 첫 줄                                 |
| ----------- | ------------------------------------------ |
| 실패담      | "일주일 만든 앱 유저 0명일 때의 기분"      |
| 불편한 진실 | "코딩 실력이랑 돈 버는 실력은 별개임"      |
| 과정 공유   | "오늘 처음으로 모르는 사람이 가입함"       |
| 비교        | "만드는 데 3개월, 홍보하는 데 3일도 안 씀" |
| 반전        | "퇴사하고 자유로워질 줄 알았는데"          |
| 질문        | "혼자 만들 때 제일 힘든 게 뭐임?"          |

## 타이밍

- **평일 점심 (12-13시)**: 직장인 피드 확인 시간
- **평일 저녁 (21-23시)**: 퇴근 후 소셜 타임
- **주말 오전 (10-12시)**: 여유롭게 피드 탐색
- 발행 후 **30분 내 engagement**가 도달을 결정한다. 주변에 미리 공유.

## 안티패턴

1. **한 글에 다 담기**: 문제 + 솔루션 + CTA + 링크를 한 글에 → 광고
2. **첫 글부터 링크**: 도달 감소 + "홍보구나" 인식
3. **너무 겸손**: "아직 부족하지만..." → 확신이 없으면 왜 봐야 하나
4. **너무 길게 설명**: 읽는 사람은 3초를 준다. 설명은 리플라이에서
5. **댓글에 반응 안 하기**: 초기 engagement가 생명. 모든 댓글에 답글

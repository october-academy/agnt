# 랜딩페이지 디자인 가이드

단일 HTML 랜딩페이지를 프로덕션 수준으로 생성할 때 참조하는 디자인 원칙입니다.
(Based on anthropics/claude-code frontend-design skill)

## 핵심 원칙

생성하는 랜딩페이지는:
- 기능적으로 완전하고 동작해야 함
- 시각적으로 인상적이며 일관된 미학적 비전이 있어야 함
- 모든 디테일이 정교해야 함

**절대 원칙**: 명확한 컨셉 방향을 잡고 정밀하게 실행하라. 대담한 맥시멀리즘이든 정제된 미니멀리즘이든 둘 다 좋다 — 핵심은 의도성이지 강도가 아니다.

## 코드 작성 전 사고

HTML을 쓰기 전에 반드시 생각할 것:
1. 이 랜딩페이지는 누구를 위한 것인가? (state.json의 character.target 참조)
2. 어떤 미학적 방향이 맞는가? (미니멀, 브루탈리즘, 레트로-퓨처, 오가닉 등)
3. 프로젝트의 성격과 타겟에 맞는 톤은? (전문적, 장난기, 미래지향적, 따뜻한)
4. 무엇이 이 디자인을 기억에 남게 만드는가?

## 타이포그래피

아름답고, 독특하고, 흥미로운 폰트를 선택하라. 기본 폰트(Arial, Inter)를 피하고 차별화된 선택을 하라.

**구현 방법**:
- Google Fonts CDN `<link>` — 유일하게 허용하는 외부 의존성
- 디스플레이 폰트(헤드라인) + 본문 폰트(설명) 페어링
- 프로젝트 분위기에 맞는 폰트 선택 (교육→신뢰감, SaaS→현대적, 커뮤니티→따뜻한)

**좋은 예시**:
- 한국어: Pretendard + Noto Sans KR, Wanted Sans + IBM Plex Sans KR
- 영문 혼용: Space Grotesk + Inter, Outfit + DM Sans, Plus Jakarta Sans + Source Sans 3

## 컬러 & 테마

CSS 변수로 일관된 컬러 팔레트를 구축하라:
- 분위기를 설정하는 도미넌트 컬러
- 인터랙티브 요소용 샤프한 액센트 컬러
- 접근성을 고려한 대비 비율

**금지**: 뻔한 보라색 그라디언트, 의미 없는 배색. 프로젝트 맥락과 목적에 맞는 컬러를 선택하라.

**구현**:
```css
:root {
  --color-bg: ...;
  --color-surface: ...;
  --color-text: ...;
  --color-text-secondary: ...;
  --color-accent: ...;
  --color-accent-hover: ...;
}
```

## 모션 & 애니메이션

CSS-only 솔루션을 우선하라. 고임팩트 순간에 집중:
- `@keyframes`로 페이지 로드 시 순차적 등장 (stagger)
- 의미 있는 hover 상태
- 부드러운 상태 전환

**구현 팁**:
- `animation-delay`로 섹션별 순차 등장
- `transform` + `opacity`로 성능 좋은 애니메이션
- `transition`으로 hover/focus 피드백

## 공간 구성

뻔한 그리드 레이아웃을 넘어서라:
- 비대칭 배치로 시각적 흥미 유발
- 오버랩과 레이어링의 전략적 사용
- 넉넉한 여백 OR 통제된 밀도 (둘 다는 안 됨)

## 시각적 디테일

분위기를 더하는 요소:
- 미묘한 그라디언트와 컬러 전환
- 맥락에 맞는 텍스처와 패턴 (CSS `background-image`로 가능)
- 그림자와 깊이 효과
- 프로젝트에 특화된 장식 요소

## 금지 사항 — Generic AI 미학

- 남용되는 폰트 (Inter, Poppins 도배)
- 생각 없는 뻔한 배색 (보라 그라디언트 + 카드 레이아웃)
- 예측 가능한 카드 기반 레이아웃
- 맥락과 캐릭터가 없는 디자인
- 모든 섹션이 같은 패딩 + 같은 구조
- Hero → Cards → CTA 공장 찍어내기 패턴

## 랜딩페이지 특화 가이드

### 구조 (필수 섹션)
1. **Hero**: 핵심 문제를 고객 관점으로 한 문장 (헤드라인) + 솔루션 가치 (서브) + CTA
2. **Problems**: 인터뷰에서 발견한 불편함 3가지 — 공감을 유도하는 구체적 묘사
3. **Solution**: 어떻게 해결하는지 — 기능이 아닌 결과/가치 중심
4. **Social Proof / Numbers**: 규모감이나 비용 절감 수치 (인터뷰 데이터 활용)
5. **Bottom CTA**: 반복된 행동 유도
6. **Footer**: 프로젝트명, 간단 소개

### 카피라이팅 원칙
- 헤드라인은 **고객의 고통**을 직접 언급 (솔루션 설명 아님)
- 서브 헤드라인은 **해결 후 상태**를 묘사
- 문제 섹션은 "당신도 이런 경험 있죠?" 공감 유도
- CTA는 구체적 행동 ("이메일 입력하기" > "시작하기")

### 기술 제약
- 단일 HTML 파일 (인라인 `<style>`)
- 외부 의존성: Google Fonts CDN만 허용
- 모바일 반응형 (`clamp()`, `min()`, media queries)
- 한국어 콘텐츠
- JS는 최소한 (폼 제출 alert 정도)
